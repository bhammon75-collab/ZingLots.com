# Migration File Naming Guide

## Current State
The migrations directory contains 35 migration files with non-descriptive names like `20250812181614-.sql`.

## Recommended Naming Convention

### Format
```
{timestamp}-{descriptive-name}.sql
```

Where:
- `{timestamp}` - Keep the existing timestamp (YYYYMMDDHHMMSS)
- `{descriptive-name}` - Add a kebab-case description of what the migration does

### Examples
- `20250812181614-initial-schema.sql`
- `20250812181717-create-app-schema.sql`
- `20250812183435-add-auth-helpers.sql`
- `20250812222928-create-lots-table.sql`
- `20250813173907-add-bid-tracking.sql`

## Safe Migration Renaming Process

**IMPORTANT**: Never rename migrations that have already been applied to production!

### For Local Development Only:
1. Check which migrations have been applied:
   ```bash
   supabase migration list
   ```

2. Only rename migrations that haven't been applied to production

3. Use this script to safely rename with preserved timestamps:
   ```bash
   # Example rename script
   mv 20250812181614-.sql 20250812181614-initial-schema.sql
   ```

## Going Forward

For all new migrations:
1. Use `supabase migration new {descriptive-name}`
2. This will automatically create: `{timestamp}-{descriptive-name}.sql`
3. Keep descriptions short but meaningful (3-5 words)
4. Use kebab-case for consistency

## Benefits
- Easier debugging when migrations fail
- Better understanding of database evolution
- Simpler rollback identification
- Improved team collaboration

## Note on Safety
The migrations with UUIDs (last 4 files) appear to be generated by a tool and should be left as-is. Only rename the dash-only migrations if they haven't been applied to production.
